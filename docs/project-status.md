## Правила ведения проекта

### 1. Сначала план, потом реализация

Не прыгать сразу в код, пока не понятен текущий этап.

### 2. Делать маленькими завершёнными кусками

Не пытаться делать весь проект одним заходом.

### 3. Держать фокус на MVP

Любая новая идея проверяется вопросом:

**это нужно для первой рабочей версии или нет?**

### 4. Не полировать раньше времени

Красота UI, идеальная архитектура и расширения — только после рабочего сквозного сценария.

### 5. Оценивать результат по критерию «работает / не работает»

На этом проекте особенно важен именно рабочий результат.

## Текущее состояние проекта

- Этап 0 завершён: границы MVP зафиксированы.
- Этап 1 завершён: контрактный этап закрыт.
- Артефакты этапа: `contracts/typespec/main.tsp`, `contracts/openapi/openapi.yaml`.
- Добавлена минимальная инфраструктура TypeSpec/OpenAPI: `package.json`, `package-lock.json`, `tspconfig.yaml`.
- Этап 2 завершён: технический каркас собран, структура репозитория определена, локальный запуск backend и frontend описан.
- Docker не блокирует завершение Этапа 2 и остаётся отдельным более поздним инфраструктурным этапом.
- Этап 3 завершён: backend MVP реализован по контракту с in-memory хранением.
- Этап 4 завершён: frontend MVP реализован для публичного flow и owner-раздела.
- Этап 5 завершён: сквозная интеграция подтверждена, основной сценарий проходит локально end-to-end.
- Этап 6 завершён: Playwright e2e покрывают happy path и негативные сценарии, локальный запуск воспроизводим одной командой.
- Этап 7 завершён: CI запускает проверки контракта, backend, frontend build и Playwright e2e на push/pull_request.
- SonarQube Cloud подключён к существующему CI в режиме CI-based analysis.
- Этап 8 завершён: проект упакован в Docker через один корневой `Dockerfile`, локальный контейнерный запуск воспроизводим.
- Следующий этап: Этап 9 — deploy.
